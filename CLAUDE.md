# XIVPedia プロジェクト規約

## プロジェクト概要

- **リポジトリ**: `Umisyo/XIVPedia`
- **説明**: ユーザー投稿型のFF14攻略サイト

## 開発ワークフロー

### セッション開始ルール

- セッション開始時は必ず現在のブランチを確認し、main ブランチでなければ main に移動し、最新化すること。

1. Issue を作成し、適切なラベルを付与
2. ブランチを作成して実装開始
3. 実装完了後 PR を作成
4. レビュー・承認後マージ
5. Issue をクローズ

### Worktree ルール

- コード変更時は**必ず**ブランチ + worktree を作成すること（`/dev start <issue-number>`）
- **main ブランチで直接コード変更を行うことは禁止**
- 全作業（編集・テスト・コミット）は worktree 内で実施する
- Worktree 配置先: `../worktrees/<branch-name>`（リポジトリの親ディレクトリ基準）

### エージェントチーム

- 複雑な実装は Task ツールでエージェントチームを並列起動する（`/dev team`）
- 同一ファイルの競合回避のため、各エージェントの担当ファイルを明確に分離する
- エージェント数は 2-4 が適切

### PR・レビュールール

- 実装完了後は必ず PR を作成し、本文に `Closes #<issue-number>` を含めて Issue と紐づけること
- PR タイトルはコミットメッセージ規約に準拠: `<type>: <description> (#<issue-number>)`
- **PR 作成前に必ず `codex review --base main` を実行してコードレビューを通すこと**
- レビューで指摘があれば修正してから PR を作成する
- `/dev team` でエージェントチームを起動する場合、**必ずレビュー担当エージェントを含めること**（レビューステップを飛ばさないため）
- ブラウザでの動作確認が必要な場合は `agent-browser` コマンドを使用する

## ブランチ命名規則

```
<type>/<short-description>
```

**例**: `feat/user-auth`, `fix/login-error`, `docs/api-reference`

**type 一覧**:
- `feat` — 新機能
- `fix` — バグ修正
- `docs` — ドキュメント
- `chore` — 雑務・メンテナンス
- `refactor` — リファクタリング

## コミットメッセージ規約

```
<type>: <description> (#<issue-number>)
```

**例**: `feat: ユーザー認証機能を追加 (#12)`

**type 一覧**: `feat`, `fix`, `docs`, `chore`, `refactor`, `test`, `ci`

## Issue テンプレート

```markdown

## モチベーション

<!-- どのように実現するか -->

## 受け入れ条件
- [ ] 条件1
- [ ] 条件2

## 備考
<!-- 参考情報など -->
```

<!-- 何が問題か -->

## 実際の動作
<!-- 実際にどうなっているか -->

- ブラウザ:
- バージョン:

<!-- スクリーンショット、ログなど -->
